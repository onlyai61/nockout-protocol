# KNOCKOUT PROTOCOL 전체 게임 와이어프레임 v0.1

작성일: 2026-05-20 KST  
목적: UX/UI 팀이 전체 게임 화면 구조를 한 번에 잡고, 이후 화면별 상세 시안을 나눠 작업하기 위한 기준안

---

## 1. 전체 UX 방향

KNOCKOUT PROTOCOL은 **로봇을 조립하고, 아레나에서 파츠를 부수고, 코어가 열린 순간 KO를 넣는 1v1 액션 게임**으로 잡는다.

UX의 핵심은 다음 순서가 끊기지 않게 만드는 것이다.

```text
기체 선택 → 격납고 조립 → 출격 → 전투 → 파츠 파괴 → 코어 노출 → KO → 보상 회수 → 재정비 → 재출격
```

### 핵심 감정선
- 선택: “내 로봇을 고른다.”
- 소유: “파츠를 바꾸니 성능이 달라진다.”
- 긴장: “전투 중 파츠가 부서진다.”
- 기회: “상대 코어가 열렸다.”
- 쾌감: “KO 프로토콜 발동.”
- 반복: “회수한 파츠로 다시 강해진다.”

---

## 2. 메인 화면 플로우

### 0) 스플래시 / 로딩
**목적:** 게임 진입 전 브랜드 톤 고정

화면 구성:
- 중앙: `KNOCKOUT PROTOCOL` 로고
- 하단: 로딩 바 / 빌드 버전
- 배경: 어두운 정비창, 스파크, 로봇 실루엣

다음 화면: 타이틀

---

### 1) 타이틀 / 시작 화면
**목적:** 첫 인상 + 빠른 진입

화면 구성:
- 중앙: 대표 로봇 또는 아레나 키비주얼
- 좌측: 메뉴
  - 계속하기
  - 새 게임
  - 격납고
  - 설정
- 우측: 최근 전투 기록 / 오늘의 KO 조건 / 테스트 빌드 안내
- 하단: `START` 메인 CTA

UX 메모:
- 버튼은 1개를 가장 강하게 보여준다.
- AI 생성 이미지 느낌보다 “게임 UI로 구현 가능한 HUD” 느낌을 우선한다.

---

### 2) 로비 / 정비소 홈
**목적:** 플레이어의 현재 상태와 다음 행동을 알려주는 허브

화면 구성:
- 중앙: 현재 장착 로봇 3D/2.5D 프리뷰
- 좌측: 주요 메뉴
  - 출격
  - 격납고
  - 파츠 보관함
  - 전투 기록
  - 설정
- 우측: 현재 로봇 상태
  - 코어 내구도
  - 장갑
  - 이동성
  - 열/에너지
  - 손상 파츠
- 하단: 추천 행동
  - `손상 파츠 수리하기`
  - `새 파츠 장착하기`
  - `바로 출격`

다음 화면:
- 출격 → 매칭/로딩
- 격납고 → 파츠 조립

---

### 3) 기체 선택
**목적:** 플레이 성향 선택

기본 기체 3종:
- Bruiser: 느리지만 강한 근접형
- Skimmer: 빠르고 회피 중심
- Bulwark: 방어/버티기 중심

화면 구성:
- 중앙: 선택 기체 대형 프리뷰
- 좌측: 기체 리스트 카드
- 우측: 스탯 비교
  - Power
  - Armor
  - Speed
  - Heat Control
  - KO Stability
- 하단: `선택`, `테스트 보기`, `뒤로`

UX 메모:
- 수치만 보여주지 말고 “플레이 방식 한 줄 설명”을 함께 붙인다.
- 신규 유저는 추천 태그를 표시한다.

---

### 4) 격납고 / 파츠 조립
**목적:** 게임의 핵심 루프인 조립과 성능 변화를 체감시키는 화면

화면 구성:
- 중앙: 로봇 프리뷰
- 좌측: 파츠 카테고리
  - Core
  - Head/Sensor
  - Arm L/R
  - Weapon
  - Armor
  - Legs/Wheels
- 우측: 선택 파츠 상세
  - 성능 변화
  - 장점/단점
  - 무게
  - 열 발생
  - 파괴 시 패널티
- 하단: 장착 슬롯 바 + CTA
  - `장착`
  - `비교`
  - `수리`
  - `출격 준비`

UX 메모:
- 파츠를 바꾸면 중앙 로봇 외형과 우측 스탯이 즉시 바뀌어야 한다.
- “좋은 파츠 = 무조건 강함”이 아니라 무게/열/속도 트레이드오프를 보여준다.

---

### 5) 매칭 / 전투 로딩
**목적:** 전투 전에 상대, 맵, KO 조건을 이해시키는 화면

화면 구성:
- 중앙: 내 로봇 VS 상대 로봇
- 좌측: 내 빌드 요약
- 우측: 상대 예상 약점
- 하단: 조작 힌트 / KO 조건

표시 정보:
- 맵 이름
- 제한 시간
- 상대 주요 무기
- 상대 약점 파츠
- KO 조건 예시: `상대 코어 노출 3초 이내 KO 입력`

---

### 6) 전투 HUD
**목적:** 실제 플레이 중 가장 중요한 정보만 보여준다.

화면 구성:
- 중앙: 전투 시야, 최대한 비워둔다.
- 상단 중앙: 타이머 / 라운드 / 경고 상태
- 상단 좌측: 내 코어/장갑/에너지
- 상단 우측: 상대 코어/장갑/파츠 손상도
- 좌하단: 이동 조이스틱
- 우하단: 공격 / 회피 / 특수 / KO 버튼
- 좌측 미니 패널: 내 파츠 손상 로그
- 우측 미니 패널: 상대 파츠 파괴 로그

전투 정보 우선순위:
1. 지금 조작해야 하는 것
2. 코어 노출 여부
3. 파츠 파괴/손상 상태
4. 열/에너지 위험
5. 보조 로그

UX 메모:
- 중앙 전투 시야를 UI가 가리지 않게 한다.
- KO 가능 상태는 색만 바꾸지 말고 사운드/진동/텍스트/버튼 상태까지 함께 바뀌어야 한다.

---

### 7) 파츠 파괴 / 코어 노출 알림
**목적:** 전투 중 승부의 흐름이 바뀌는 순간을 명확히 알린다.

상태 예시:
- `LEFT ARM DAMAGED`
- `ENEMY ARMOR BROKEN`
- `CORE EXPOSED`
- `KO WINDOW OPEN`

화면 구성:
- 중앙 근처: 짧은 경고 텍스트
- 상대 위: 코어 노출 아이콘
- KO 버튼: 비활성 → 충전 → 활성
- HUD 색상: 시안/주황 → 위험 시 레드

---

### 8) KO 프로토콜 발동 화면
**목적:** 게임의 대표 쾌감 구간

화면 구성:
- 순간 슬로우 모션
- 중앙: `KNOCKOUT PROTOCOL` 경고 프레임
- 하단/우하단: KO 입력 버튼 또는 타이밍 게이지
- 배경: 글리치, 스캔라인, 카메라 줌

UX 메모:
- 과하게 복잡한 QTE보다는 “지금 끝낼 수 있다”가 즉시 읽히는 연출이 우선이다.
- 성공/실패 피드백이 즉각적이어야 한다.

---

### 9) 전투 결과 / 회수
**목적:** 승패와 다음 성장 재료를 정리한다.

화면 구성:
- 중앙: 승리/패배/KO 하이라이트
- 좌측: 전투 통계
  - 전투 시간
  - 준 피해
  - 받은 피해
  - 파괴한 파츠
  - KO 여부
- 우측: 회수 보상
  - 고철
  - 데이터 칩
  - 파츠 조각
  - 신규 파츠
- 하단: CTA
  - `재정비`
  - `다시 매칭`
  - `하이라이트 보기`

UX 메모:
- 패배해도 “무엇을 고치면 되는지”가 보여야 한다.
- 보상은 카드 형태로 명확히 보여준다.

---

### 10) 재정비 / 수리 추천
**목적:** 결과를 다음 플레이로 연결한다.

화면 구성:
- 중앙: 손상된 로봇
- 좌측: 손상 파츠 리스트
- 우측: 추천 조치
  - 수리
  - 교체
  - 빌드 변경
- 하단: `추천 수리 적용`, `직접 조립`, `재출격`

UX 메모:
- 유저가 결과 화면에서 이탈하지 않고 자연스럽게 격납고로 돌아오게 만든다.
- 초반에는 자동 추천을 강하게 제공한다.

## 4. 화면 작업 우선순위

### 1차 MVP
- 타이틀
- 로비/정비소 홈
- 격납고/파츠 조립
- 전투 HUD
- KO 발동
- 전투 결과

### 2차
- 기체 선택
- 매칭/로딩
- 재정비 추천
- 파츠 보관함
- 전투 기록

### 3차
- 설정
- 튜토리얼 상세
- 하이라이트 다시보기
- 랭킹/시즌/온라인 요소

---

## 5. UX/UI 팀 작업 요청

이번 단계에서 UX/UI 채널에서 잡아야 할 것:

1. 위 10개 화면 중 **1차 MVP 화면 6개**를 먼저 와이어프레임으로 확정
2. 모바일/PC 중 우선 기준은 **가로 16:9 전투 화면**으로 통일
3. 전투 HUD는 중앙 시야를 비우고, 좌하단 조이스틱 / 우하단 전투 버튼 구조를 기본값으로 사용
4. KO 가능 상태의 시각 언어를 별도 정의
   - 색
   - 버튼 상태
   - 경고 문구
   - 게이지
   - 연출
5. 개발팀이 바로 구현할 수 있게 컴포넌트 단위로 쪼개서 전달

---

## 6. 한 줄 결론

KNOCKOUT PROTOCOL의 UX는 “멋진 SF 화면”보다 **조립 → 파괴 → 코어 노출 → KO → 회수 → 재정비**가 한 번에 읽히는 구조가 우선이다.
